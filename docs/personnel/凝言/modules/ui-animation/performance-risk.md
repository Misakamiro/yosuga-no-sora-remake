# 动画性能风险

## 概要

只要复用现有 `Sprite` activation 路径，并限制动画作用在少量父级/分组层上，UI 动画升级可以保持低风险。最大的性能风险不是简单透明度或位置插值，而是粒子/环境效果、大量子控件同时动画、全屏重绘，以及大面积规则遮罩转场。

## 当前运行成本模型

`Sprite.tjs`：

- 每个活动中的 `Sprite.beginActivation()` 会把 `on_activationTimer` 注册到 `System.addContinuousHandler`。
- 动画完成前，每个活动精灵都会更新位置、透明度、缩放或旋转。
- 成本随“正在动画的精灵数”增加，而不是随所有可见精灵增加。

`AnimationSequence.tjs`：

- `.ams` 序列可能创建多个 `AnimationSprite`。
- `motion` 使用每个点 7 个参数的样条路径插值：x、y、z、透明度、zoomx、zoomy、旋转。
- `particle` 可创建多个精灵实例并同时动画。

`Action.tjs`：

- `ActionList.doAction()` 每 tick 遍历活动动作序列。
- wave/random/shock/quake 对单个对象较轻，但应用到很多 UI 子控件时会变重。

`EnvEffect.tjs`：

- 雪和雨使用定时器和逐对象更新。
- 雨会用 `fillRect` 重绘一层，再逐个绘制雨滴。
- 雪/雨有旧生成范围，`1920x1080` 下如果与复杂 UI 同屏，需要重新测试。

`rule/*`：

- 规则遮罩视觉上有用，但 `1920x1080` 全屏转场的像素处理成本明显高于 `800x600`。

## 风险表

| 风险 | 严重度 | 原因 | 缓解方式 |
| --- | --- | --- | --- |
| 给设置/存档/读档里的每个按钮单独动画 | 高 | 大窗口可能包含大量控件或卡片，每个都有活动 handler/action。 | 动画父级组或页面表面，不动画每个控件。 |
| 普通 UI 操作用全屏规则转场 | 高 | `1920x1080` 遮罩转场会处理大面积像素，在 KRKRZ 下可能卡顿。 | 规则转场只用于场景切换；UI 用淡入/滑动。 |
| 菜单粒子装饰 | 高 | `AnimationSequence.particle` 会创建多个精灵和样条动画。 | 普通 UI 禁用；如只用于标题，限制数量并在退出时停止。 |
| 环境效果叠加 UI 重动画 | 中 | 雪/雨已有连续定时器和大量移动对象。 | 未运行测试前，不要在重环境效果场景中叠加大量 UI 动画。 |
| 动画期间长时间禁用点击区 | 中 | 现有代码会切换 `hitThreshold`；回调出错可能导致 UI 不可点击。 | 时长保持短，完成回调恢复点击状态，增加快速开关测试。 |
| 机械缩放坐标 | 中 | 位移可能变得过大，重绘区域增加且观感差。 | 按用途调整：子菜单 `8-16px`，对话框 `18-28px`，标题组切换 `42-56px`。 |
| 对旧低分辨率 UI 做缩放/旋转 | 中 | 更容易产生视觉瑕疵，也可能增加变换成本。 | 优先透明度和位置；缩放只用于标题装饰。 |
| 多层嵌套转场 | 中 | 父级和子级透明度/位移叠加后难取消。 | 一个转场只设一个 owner，启动前停止或刷新旧动画。 |
| 子菜单 idle trigger 叠加动画 | 低/中 | `SubMenu` 使用 `AsyncTrigger` 处理进出检查，动画可能延迟状态变化。 | 子菜单显示/隐藏保持短，输入状态显式管理。 |

## 推荐限制

第一版实现时：

- 标题入场：同时动画不超过 3-5 个组/精灵。
- 对话框：动画消息框/底板组，不动画每个文字或按钮子项。
- 系统菜单：动画菜单区域/组，不动画全部 9 个按钮。
- 子菜单：只动画子菜单容器。
- 选项：只动画可见选项，错开显示总时长不超过 `320ms`。
- 设置/存档/读档：动画一个页面/网格容器，第一版不要单独动画每个滑条、标签、存档卡。

## 跳过 / 关闭特效行为

保留快速路径：

- 屏幕效果关闭时，UI 动画应立即完成或使用 `0ms`。
- 跳过模式下，对话框、脸图窗口、光标转场不能拖慢文本流程。
- 快速点击和键盘导航不应排队多个互相冲突的转场。

## 验证清单

实施后运行：

1. 标题待机、开始、读档/设置、Bonus 进入/退出。
2. ADV 文本显示/隐藏和对话框类型切换。
3. 角色脸图切换、同角色表情切换、隐藏/显示。
4. 系统菜单锁定/解锁、鼠标移出、子菜单打开/关闭。
5. 选项列表显示、键盘焦点、鼠标悬停、选择。
6. 设置/存档/读档打开/关闭和标签/翻页切换。
7. 如可达，重复测试 `CONFIG.screenEffect == 0`。
8. 相关流程在跳过/自动模式下重复测试。

需要收集的运行证据：

- 多个 UI 组同时动画时是否卡顿。
- 快速打开/关闭后输入是否仍响应。
- 是否留下活动精灵或不可见但可点击区域。
- `1920x1080` 规则转场是否有视觉瑕疵或明显耗时峰值。

## 本轮结果

本轮只做动画方案和文档，没有做运行时性能分析。静态审查结论是：低风险路线应复用现有动画基础，不新增定时器，不做大范围子控件动画。
