# 性能优化计划

## 原则

先优化低风险层。每一步都要有改动前/后的测量和截图。不要把性能修改与分辨率迁移、资源替换、剧本修改混在同一批。

## 推荐顺序

### 1. 先采集基线

改代码前记录：

- 启动到标题耗时。
- 标题待机 CPU 和内存。
- 无环境效果的 ADV 场景 CPU 和内存。
- 雪效果 ADV 场景。
- 雨效果 ADV 场景。
- 如果能触发，记录背景滚动场景。
- 角色切换频繁的转场场景。
- BGM 交叉淡入加语音加 ENVSE/BGV。
- 存档/读档截图路径。

原因：

- 静态检查只显示多个可能风险，无法判断当前构建中哪个最严重。

### 2. 兼容稳定后清理调试输出

候选修改：

- release/performance 构建中把 `tvpwin32.cf` 的 `debugwin` 从 yes 改为 no。
- 用本地 debug 标志控制启动和调试消息。

低风险原因：

- 不改变视觉行为和剧本语义。

不要在以下情况做：

- 团队仍在调查启动链或兼容崩溃。

### 3. 环境效果限速和暂停策略

候选修改：

- 保持效果间隔可由脚本配置，但文档中给出推荐最小值。
- 只有在视觉对比后，才提高默认基础/效果间隔。
- ADV 层隐藏、失活或替换时停止或暂停效果。
- 如果 trace 证明有加载抖动，再考虑避免每次停止/开始都销毁和重载雪图。

优先原因：

- 环境效果是最明显的 timer 驱动成本。

风险：

- 雨/雪运动可能肉眼卡顿。
- 剧本演出可能依赖当前视觉密度。

### 4. 动画连续 handler 卫生检查

候选修改：

- 如果 KRKRZ 能安全检测 handler 状态，可在添加连续 handler 前做重复注册防护。
- 如果 profiling 证明缩放插值计算昂贵，再避免每 tick 重算辅助状态。
- 测试版增加活动 sprite activation / action callback 计数。

优先原因：

- 有助于发现失控动画，不立即改变资源。

风险：

- handler 注册语义对引擎敏感，只能用小测试用例验证后再改。

### 5. 降低本地包装层重复图片加载

候选修改：

- 在 `Sprite.loadImages` 中，只有当 storage 名称、层状态相同，且调用方不依赖重置副作用时，才跳过相同文件重载。
- `EnvEffectBgScroll.setParam` 中，如果 `_param.file` 未变且只改运动/cycle，可跳过重载。
- 雪效果可懒创建一次，并在所属 `ADVScreen` 生命周期内保留小图。

放到后面原因：

- 图片缓存很容易出错，因为加载可能重置尺寸、透明度、仿射偏移或 cell 状态。

风险：

- 同名文件可能因资源路径变化而需要重载。
- 层设置的副作用可能消失。

### 6. 音频 buffer 清理调优

候选修改：

- 为 BGM、SE、ENVSE、VOICE、BGV 添加 `SoundLayer.count` 诊断记录。
- 确认 fade 最终会把 buffer 标记为 destroy。
- 只有证据显示 stop 后 buffer 长时间偏高时，才缩短清理或在无淡出 stop 后立即清理。

放到后面原因：

- 当前 `1000ms` 清理 timer CPU 不高。
- 错误清理可能切断淡出或循环环境音。

### 7. 分辨率分支存在后再做 1920x1080 专项优化

候选修改：

- 测量 `1920x1080` 下全画布 `piledCopy` 和转场耗时。
- 决定是否需要复用转场缓冲、缩小 dirty region 或使用低成本转场。
- 检查 1080p 画布下雨效果填充和背景滚动循环。

放到最后原因：

- 当前运行时仍是 `800x600`。过早做 1080p 优化会掩盖兼容问题。

## 当前不建议做

- 不要全局降低 `contfreq`，否则可能影响输入、转场和动画流畅度。
- 不要从剧本中删除环境效果。
- 不要在没有视觉验收前替换全屏转场行为。
- 不要改变 BGM 交叉淡入双缓冲。
- 不要在本轮修改图片资源、剧本脚本、封包文件或 `1920x1080` 迁移常量。

## 未来优化完成标准

- 每次只改一类风险。
- 保存改动前/后的测量。
- 为视觉效果保存截图或视频证据。
- 启动无回归。
- 音频停止/重启无缺失。
- 场景切换后没有卡住的动画或连续 handler。
- 存档/读档、自动/跳过、窗口/全屏流程仍通过。
